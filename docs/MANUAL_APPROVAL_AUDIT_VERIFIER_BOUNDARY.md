# Manual Approval Audit Verifier Boundary (Phase 6G)

## Purpose

Phase 6G defines the contract for a **future** audit verifier that checks audit
artifacts produced by a future single-gate manual approval wrapper. It defines
the boundary only; it implements no audit verifier and executes nothing.

## Scope

- boundary-only; docs/tests/task-only
- no runtime verifier
- no runtime command
- no vault read/write
- no primitive execution
- no approval mutation
- no chain execution
- no global approve

Primitive file names and approval flag names are referenced as names only; this
document contains no command forms and no approval flag assignments.

## Audit verifier purpose

A future audit verifier must:

- verify audit artifacts generated by a future single-gate wrapper
- ensure the selected gate was single-gate only
- ensure `mutation_attempted` is explicit
- ensure `outcome` is one of `success`, `failure`, `blocked`, `prevented`
- ensure the audit references the Phase 6B packet, the Phase 6C verifier, and
  the Phase 6E execution plan
- ensure there is no hidden chain execution
- report evidence only
- remain read-only

## Required audit fields

The future audit artifact must include:

- `product_id`
- `report_week`
- `selected_gate`
- `primitive_name`
- `operator`
- `approval_reason`
- `timestamp`
- `source_packet_path`
- `verifier_path`
- `execution_plan_path`
- `precondition_summary`
- `result_summary`
- `outcome`
- `mutation_attempted`
- `gate_specific_approval_intent`
- `approved_flag_name`
- `wrapper_version`
- `audit_schema_version`

Phase 6G is a verifier-level **superset** of the Phase 6F audit model. The Phase
6F document is not rewritten here beyond an additive pointer.

## Audit verifier rules

- `selected_gate` must be one of `promote`, `decision`, `finalization`.
- `primitive_name` must match the selected gate:
  - promote -> `promote_product_candidates.py`
  - decision -> `create_decision.py`
  - finalization -> `finalize_decision.py`
- `approved_flag_name` must match the selected gate:
  - promote -> `APPROVE_PROMOTE`
  - decision -> `APPROVE_DECISION`
  - finalization -> `APPROVE_FINALIZE`
- only one gate per audit artifact
- no multi-gate list
- `outcome` must be one of `success`, `failure`, `blocked`, `prevented`
- `blocked` or `prevented` means `mutation_attempted` must be false
- `success` or `failure` means `mutation_attempted` may be true, but must be
  explicit
- the audit must reference the Phase 6B packet path
- the audit must reference the Phase 6C verifier path
- the audit must reference the Phase 6E execution plan path
- all referenced paths must be relative `tmp/` paths
- referenced paths must not be absolute
- referenced paths must not include traversal
- referenced paths must not include vault paths
- the audit must not contain approval command forms
- the audit must not contain external URLs
- the audit must not contain secrets
- the audit must not contain operator-local paths

## Single-gate consistency

A future audit verifier must flag these as invalid:

- multi-gate list
- approve-all
- global approval
- automatic next-gate
- chain execution
- hidden promotion
- UI-direct approval
- selected gate mismatch
- primitive mismatch
- approval flag mismatch

## Future verifier verdict policy

```text
valid:
- all required fields are present
- all consistency checks pass
- no leakage
- no unsafe paths
- no command forms

warning:
- optional metadata is missing
- referenced artifacts are stale
- referenced artifact hash mismatch in a future implementation
- all hard safety checks still pass

invalid:
- required field missing
- gate mismatch
- primitive mismatch
- flag mismatch
- mutation state inconsistency
- unsafe path
- leakage
- command form present
- hidden chain execution
- multi-gate evidence
```

Future exit policy: `valid` exits 0, `warning` exits 0, `invalid` exits
non-zero. This is only a future verifier contract; Phase 6G does not implement
the verifier.

## Failure and safety model

- the audit verifier never executes primitives
- the audit verifier never mutates the vault
- the audit verifier reports evidence only
- the audit verifier must fail closed on unsafe audit content
- the audit verifier must make partial completion explicit
- unsafe content means invalid
- command forms mean invalid
- unsafe paths mean invalid

## Forbidden automation

The following are forbidden behaviors, not commands to run:

- autopublish
- campaign launch
- affiliate link generation
- marketplace connector
- external API submit
- hidden promotion
- UI-direct approval
- global approve
- multi-gate execution
- approve-all mode
- automatic next-gate execution
- finalization without compliance approved
- vault write without explicit gate-specific approval
- backend / API / database
- network calls

## Future roadmap

- Phase 6H may update the release snapshot.
- An actual audit verifier implementation must be a separate, explicitly
  approved phase.
- An actual single-gate wrapper implementation must be a separate, explicitly
  approved phase.
- Any backend, API, database, or marketplace implementation must be a separate
  approved phase.

## Known limitations

- Boundary documentation only; no audit verifier, command, or mutation exists.
- Operator identity is a placeholder; no authentication exists.
- Future Phase 6H+ are separate implementation phases under their own approval.
