# Task 040 - Phase 6G Manual Approval Audit Verifier Boundary Plan

## 1. Purpose

Define the contract for a future audit verifier that checks audit artifacts
produced by a future single-gate manual approval wrapper. Phase 6G defines the
boundary only and implements no audit verifier.

## 2. Scope

- Author the audit verifier boundary contract and a docs-contract test.
- Additively note the boundary in the single-gate wrapper boundary doc, ROADMAP,
  and PROJECT_STATE.
- Reference primitive file names and approval flag names as names only; never
  execute, modify, or include command forms for them.

- `codex/tasks/040-phase6g-manual-approval-audit-verifier-boundary.md`
- `docs/MANUAL_APPROVAL_AUDIT_VERIFIER_BOUNDARY.md`

## 4. Audit verifier purpose

A future audit verifier must verify audit artifacts generated by a future
single-gate wrapper: ensure the selected gate was single-gate only, ensure
`mutation_attempted` is explicit, ensure `outcome` is one of `success`,
`failure`, `blocked`, `prevented`, ensure the audit references the Phase 6B
packet, Phase 6C verifier, and Phase 6E execution plan, ensure there is no
hidden chain execution, report evidence only, and remain read-only.

## 5. Required audit fields

`product_id`, `report_week`, `selected_gate`, `primitive_name`, `operator`,
`approval_reason`, `timestamp`, `source_packet_path`, `verifier_path`,
`execution_plan_path`, `precondition_summary`, `result_summary`, `outcome`,
`mutation_attempted`, `gate_specific_approval_intent`, `approved_flag_name`,
`wrapper_version`, `audit_schema_version`. Phase 6G is a verifier-level superset
of the Phase 6F audit model; the Phase 6F document is not rewritten beyond an
additive pointer.

## 6. Audit verifier rules

`selected_gate` in {promote, decision, finalization}; `primitive_name` matches
the gate (promote -> `promote_product_candidates.py`, decision ->
`create_decision.py`, finalization -> `finalize_decision.py`);
`approved_flag_name` matches the gate (`APPROVE_PROMOTE`/`APPROVE_DECISION`/
`APPROVE_FINALIZE`); one gate per artifact, no multi-gate list; `outcome` in
{success, failure, blocked, prevented}; `blocked`/`prevented` require
`mutation_attempted` false; `success`/`failure` allow it true but explicit; the
audit references the Phase 6B packet, Phase 6C verifier, and Phase 6E execution
plan paths; all referenced paths are relative `tmp/` paths (no absolute, no
traversal, no vault); the audit contains no approval command forms, external
URLs, secrets, or operator-local paths.

## 7. Single-gate consistency

A future verifier flags as invalid: multi-gate list, approve-all, global
approval, automatic next-gate, chain execution, hidden promotion, UI-direct
approval, selected gate mismatch, primitive mismatch, and approval flag
mismatch.

## 8. Future verifier verdict policy

`valid`: all required fields present, all consistency checks pass, no leakage,
no unsafe paths, no command forms. `warning`: optional metadata missing or
referenced artifacts stale/hash-mismatch while hard safety checks pass.
`invalid`: required field missing, gate/primitive/flag mismatch, mutation-state
inconsistency, unsafe path, leakage, command form present, hidden chain
execution, or multi-gate evidence. Future exit policy: `valid`/`warning` exit 0,
`invalid` exits non-zero. Phase 6G does not implement the verifier.

## 9. Failure and safety model

The audit verifier never executes primitives, never mutates the vault, reports
evidence only, fails closed on unsafe audit content, and makes partial
completion explicit. Unsafe content, command forms, and unsafe paths all mean
invalid.

## 10. Forbidden automation

Autopublish, campaign launch, affiliate link generation, marketplace connector,
external API submit, hidden promotion, UI-direct approval, global approve,
multi-gate execution, approve-all mode, automatic next-gate execution,
finalization without compliance approved, vault write without explicit
gate-specific approval, backend/API/database, and network calls are forbidden.

## 11. Future roadmap

Phase 6H may update the release snapshot. An actual audit verifier
implementation and an actual single-gate wrapper implementation must each be a
separate, explicitly approved phase, as must any backend/API/database/
marketplace implementation.

## 12. Test strategy

Deterministic docs-contract tests: task + boundary doc exist; boundary-only
scope; no Phase 6G runtime script; audit verifier purpose; the 18 required audit
fields; allowed gate values; primitive/flag references and per-gate matching;
outcome values; mutation-state consistency; 6B/6C/6E references; tmp-only path
rule and unsafe-path/leakage rejection; single-gate consistency; verdict policy
and exit rules; failure/safety model; forbidden automation; a no-execution guard
and static safety scanning only the two new Phase 6G files; ROADMAP/PROJECT_STATE
token regression.

## 13. Acceptance criteria

- Boundary doc, task, and test exist; the test passes.
- Phase 6F/6E/6D/6C/6B/6A and Phase 5E/3E tests still pass; full suite passes.
- No runtime script changed; no hardcoded operator path in scripts.
- No approval mutation, vault write, or primitive execution is introduced.
